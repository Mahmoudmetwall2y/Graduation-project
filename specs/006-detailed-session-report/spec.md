## User Scenarios & Testing *(mandatory)*

### User Story 1 - Review Patient's Current Heart State (Priority: P1)

Clinicians need viewing access to the comprehensive initial cardiovascular state assessed by Model 1 to determine exactly whether an anomaly (e.g., murmur or arrhythmia) is present.

**Why this priority**: Immediate assessment of the patient's existing vital condition is the core primary diagnostic value proposition of the system.

**Independent Test**: Can be fully tested by verifying that Model 1 outputs (heart state) prominently display with clinical confidence scores alongside the real-time recording data.

**Acceptance Scenarios**:
1. **Given** a successfully executed recording session, **When** the clinician opens the session page, **Then** they see a clear, structured "Current Heart State" section summarizing findings.
2. **Given** a normal heart state analysis, **When** viewing the report, **Then** the UI clearly contextualizes the result with a "Healthy/Normal" clinical indicator without false alarming.

---

### User Story 2 - Analyze Secondary Functional Modalities (Priority: P2)

Cardiologists must evaluate specific functional parameters (such as murmur severity/timing or supplementary feature analysis) processed by Model 2 to build a deeper clinical picture beyond simple binning.

**Why this priority**: Supplemental metrics differentiate this platform from basic heart monitors by providing actionable context.

**Independent Test**: Can be tested independently by ensuring that functional predictions properly populate a dedicated detailed widget.

**Acceptance Scenarios**:
1. **Given** Model 1 detected an anomaly, **When** assessing the report, **Then** Model 2's specific breakdown is clearly delineated in the timeline.

---

### User Story 3 - View Predictive Prognosis (Priority: P3)

The physician requires a clear visualization of the patient's extrapolated risk trajectory generated by Model 3 to inform long-term patient care strategies and immediate interventions.

**Why this priority**: Predictive analytics is the ultimate differentiator but relies strictly on the current/functional state being accurately represented first.

**Independent Test**: Verified by checking the display of a "Future Prognosis" or "Risk Assessment" card representing predictive timelines.

**Acceptance Scenarios**:
1. **Given** a complete cardiovascular analysis, **When** downloading or viewing the full report, **Then** the long-term prognosis evaluation is clearly segregated from current real-time observations to avoid misleading diagnoses.

---

### Edge Cases

- What happens when Model 1 fails to process, but Model 2 and 3 succeed?
- How does the system handle "Demo Mode" or low-confidence outputs for the predictive prognosis?
- How is the detailed report generated if the session is manually aborted early?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST organize the Session Detail UI chronologically and logically: Current State (Model 1), Functional Analysis (Model 2), and Predictive State (Model 3).
- **FR-002**: System MUST render a comprehensive "Detailed Report" view that aggregates all patient metadata, raw metrics, and AI model predictions.
- **FR-003**: System MUST provide the capability to export the structured Detailed Report to a printable/PDF format for clinical records.
- **FR-004**: System MUST visually separate AI predictive indicators (Model 3) from objective historical recording data to prevent diagnostic confusion. 
- **FR-005**: Display clinical disclaimers indicating that AI-driven predictive prognosis is advisory.

### Key Entities

- **Session Report Data**: Aggregate payload mapping the specific device telemetry, patient profile, and all layered inference outputs into a single hierarchical structure.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Clinicians can access and digest the 3-tier model summaries within 15 seconds of opening the session page.
- **SC-002**: 100% of generated PDF/print reports successfully include the chronological breakdown without overlapping UI elements.
- **SC-003**: Post-deployment surveys demonstrate an >80% satisfaction score in "completeness of clinical information" from beta users.
