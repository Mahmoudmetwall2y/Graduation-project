01 — Executive Summary

What is n8n and Why AscultiCor Needs It

n8n is a self-hostable, open-source workflow automation platform. Think of it as a visual programming environment for connecting services — similar to Zapier or Make.com, but you own it, it runs as a Docker container alongside your existing services, and it's completely free.

AscultiCor currently has several async background tasks that are either handled through fragile workarounds (GitHub Actions cron) or simply don't exist yet (push notifications for critical alerts). n8n fills this gap as an orchestration layer for everything that happens after real-time inference completes.

02 — Architecture

Where n8n Fits in the Stack

n8n sits alongside your existing services as a peer container. It connects to Supabase (for data triggers and writes) and to external services (LLM APIs, email, SMS). It does not touch the real-time MQTT → Inference data path.

Data Flow

  1. ESP32 devices stream PCG/ECG data via MQTT → Inference service runs ML models → results stored in Supabase (unchanged)
  2. n8n monitors Supabase for events: new predictions, pending LLM reports, critical alerts, device status changes
  3. n8n executes the appropriate workflow: call LLM API, send notification, generate digest, etc.
  4. Results are written back to Supabase and/or sent to external channels

06 — Boundaries

What n8n Does NOT Replace

n8n is not a replacement for your entire backend. These components must stay as-is:

Component Status Reason
MQTT Handler (mqtt_handler.py) Keep as-is Real-time binary signal ingestion at 22 kHz — n8n can't handle this volume or latency
Inference Engine (inference.py) Keep as-is TensorFlow/PyTorch model inference is compute-bound and tightly coupled to preprocessing
Signal Preprocessing (preprocessing.py) Keep as-is NumPy/SciPy heavy lifting — needs to run in the Python inference service
Next.js Frontend Keep as-is UI rendering, auth pages, session dashboard — all stays in Next.js
Supabase Auth & RLS Keep as-is Security layer — Supabase handles auth, row-level security, and org isolation
Mosquitto MQTT Broker Keep as-is Device authentication, topic routing, ACL enforcement
⚠

Rule of thumb: If the task involves real-time data (sub-second latency, binary streams, or high-frequency signals), keep it in FastAPI/MQTT. If the task is "when X happens, do Y after a delay" — that's n8n territory.

07 — Docker Integration

How n8n Joins the Docker Compose Stack

n8n runs as a single Docker container. It joins the existing asculticor-network so it can reach Supabase, the inference API, and other internal services. 

# Add to docker-compose.yml → services:

  n8n:
    image: n8nio/n8n:latest
    container_name: asculticor-n8n
    ports:
      - "127.0.0.1:5678:5678"       # Web UI — localhost only
    environment:
      - N8N_BASIC_AUTH_ACTIVE=true
      - N8N_BASIC_AUTH_USER=${N8N_USER:-admin}
      - N8N_BASIC_AUTH_PASSWORD=${N8N_PASSWORD:-changeme}
      - WEBHOOK_URL=http://n8n:5678/
      # Pass through so n8n workflows can query Supabase
      - SUPABASE_URL=${SUPABASE_URL}
      - SUPABASE_SERVICE_ROLE_KEY=${SUPABASE_SERVICE_ROLE_KEY}
    volumes:
      - n8n_data:/home/node/.n8n     # Persist workflows
    networks:
      - asculticor-network
    restart: unless-stopped

# Add to volumes:
  n8n_data:

After docker compose up -d, access the n8n visual editor at http://localhost:5678. Workflows are created in the browser — no code needed.
