1. Executive Summary

AscultiCor already has a solid real-time medical telemetry pipeline: the ESP32 firmware streams PCG and ECG over MQTT, the inference service processes those signals, Supabase stores the results, and the Next.js frontend presents sessions, devices, waveforms, and reports. That path is already doing the latency-sensitive work and should remain untouched.

n8n is best added as a peer service that handles the asynchronous work that starts after data has been captured or after the system notices an operational event. In practice, that means: queued LLM reports, alert notifications, device health checks, digests, escalation flows, and service monitoring.

Best immediate win

Move LLM report processing out of the current frontend route and GitHub Actions cron path into n8n, then call a real provider such as OpenAI from the workflow engine.

Best new capability

Turn telemetry, preflight failures, abnormal predictions, and timeouts into real notifications over Slack, email, SMS, or Telegram within seconds.

Biggest design rule

If the task is high-frequency, binary, or sub-second, keep it in FastAPI and MQTT. If it is event-driven, delayed, scheduled, or fan-out automation, it belongs in n8n.

Recommended starting point: implement the LLM report workflow first, then device-health alerts, then digests. This gives the fastest value while keeping deployment risk low.

2. Current Project Architecture

The repository already documents the system as a full-stack cardiac monitoring platform with ESP32 firmware, Mosquitto, FastAPI inference, Supabase, and Next.js. The README states the core flow explicitly and also notes that LLM reports are in demo mode by default.

ESP32 firmware -> MQTT broker (Mosquitto) -> FastAPI inference service -> Supabase (Postgres + storage + auth) -> Next.js dashboard Current async add-ons: - queued llm_reports - device telemetry - audit logs - live metrics - device bootstrap Recommended new orchestration layer: n8n -> LLM providers -> Slack / Email / SMS / Telegram -> scheduled health checks -> escalations -> daily summaries

What the firmware already does

The firmware publishes device status, heartbeats, and session control/data topics and subscribes to low-frequency control commands like start, stop, and reboot.

What the inference service already does

It subscribes to device status and session topics, writes telemetry, writes live metrics, stores predictions, creates recordings, and writes audit logs around session outcomes.

3. Where n8n Fits

n8n should sit alongside the existing services as a new container on the same Docker network. It should connect to Supabase using service-role access for workflow reads/writes, and optionally connect to Mosquitto for low-frequency operational topics only.

Area Keep in current stack Move to or add in n8n
Signal ingestion MQTT + FastAPI No
Waveform buffering and live stream MQTT handler + frontend live routes No
Model inference FastAPI inference service No
Queued report generation Currently in frontend route plus GitHub Actions Yes, high-value replacement
Device health checks and alerts Telemetry is already stored Yes, strong fit
Notification fan-out Not implemented strongly today Yes
Daily/weekly digests Not implemented Yes
Service health monitoring Endpoints exist Yes

Practical placement rule: n8n should observe and orchestrate. It should not become part of the hot data path between the device and inference.

4. Current Pain Points That n8n Solves

LLM pipeline is queued but not fully real

The code queues llm_reports, but actual processing is still driven through /api/llm?action=process-pending, and the implementation explicitly falls back to the demo template engine even when a non-demo provider is configured.

GitHub Actions is used as an external scheduler

The current queue processor is triggered every 5 minutes by a GitHub Actions workflow, which is fragile compared with running a first-party scheduler inside the same deployment stack.

Alert inbox exists, but alert generation looks incomplete

The UI and device API read device_alerts, but repo inspection did not find application code that actively inserts alert rows outside migrations. This is an inference from repository search, not a schema guarantee.

Telemetry and audit signals already exist

The good news is that status messages, heartbeats, preflight failures, warnings, inference completions, and session failures are already persisted. n8n can build on top of that immediately.

Current state Observed in code n8n improvement
LLM queue claim/retry logic lives in application code frontend/src/app/api/llm/route.ts Replace with workflow-based retry, delay, and failure routing
GitHub Actions calls queue processor every 5 minutes .github/workflows/process-llm-queue.yml Replace with internal n8n schedule or webhook trigger
LLM_PROVIDER=openai still generates demo text Explicit fallback in report generator Use n8n OpenAI node or HTTP Request with provider credentials
Alerts page reads device alerts Frontend consumes device_alerts Backfill the missing generation logic through workflows
Operational signals already stored Telemetry, live metrics, audit logs, predictions Use them as trigger inputs for downstream automation

5. Recommended Workflows

P0

Workflow 1: Real LLM Report Generation

This is the best first workflow because it replaces the most fragile async path in the project and upgrades a demo feature into a real one.

  1. Trigger when a new llm_reports row is pending, or poll every minute for pending rows.
  2. Fetch the related session, predictions, murmur severity, and device details.
  3. Construct a prompt from the stored data or reuse the existing prompt text.
  4. Call OpenAI through n8n.
  5. Update llm_reports with status, report text, structured JSON, tokens, latency, and errors.
  6. Optional: notify the user or clinician that the report is ready.

This workflow can fully replace the current GitHub Actions cron plus most of the queue-processing logic in the frontend route, while still keeping the route that inserts the initial queue row.

P1

Workflow 2: Clinical Alert Notifications

Trigger on abnormal outcomes such as PCG murmur detection, abnormal ECG classification, session preflight failure, or session timeout. Create a device_alerts row and send a message to Slack, email, Telegram, or SMS.

  • Use severity mapping: critical for session timeout plus abnormal result, warning for murmur-only, info for non-fatal warnings.
  • Include patient/session/device context in the outgoing message.
  • Deduplicate by session and alert type to avoid alert spam.
P1

Workflow 3: Device Health Monitoring

Run on a schedule every 1 to 5 minutes. Query devices and recent telemetry, then create or resolve alerts for stale last_seen_at, low battery, bad RSSI, high temperature, or rising error counts.

  • Offline: last seen older than threshold
  • Low battery: battery level under chosen threshold
  • Poor signal: RSSI below chosen threshold
  • High temperature: temperature above chosen threshold
  • Firmware drift: optional compare against target firmware version
P2

Workflow 4: Daily or Weekly Digest

Aggregate yesterday's sessions, abnormal findings, unresolved alerts, and device-health issues. Send a summary email to admins, operators, or clinicians.

  • Sessions by device
  • Normal vs abnormal ECG count
  • Murmur count and severity breakdown
  • Offline or unhealthy devices
  • Optional one-paragraph LLM summary
P2

Workflow 5: Recording Summary Enrichment

The schema includes device_recording_summaries and summary_json. n8n can populate these values periodically so the device dashboard has richer historical summaries without pushing more aggregation logic into the frontend.

P3

Workflow 6: Inference Service and Queue Ops Monitoring

Poll the inference service /health, /config, and /metrics endpoints using internal tokens. Alert when the service is degraded, buffers are stuck, or model availability changes unexpectedly.

P3

Workflow 7: Escalation Logic

When a critical alert is opened, send the first notification immediately. Then wait 15 to 30 minutes and check if the alert is still unresolved. If it is, escalate to a wider team or a second channel.

6. Trigger Strategy for This Project

n8n offers multiple ways to start workflows. For AscultiCor, not all trigger types are equally useful. The right choice depends on the workload and how much coupling you want between the app and the workflow engine.

Trigger type Recommended use here Why Notes
Schedule Trigger Yes, default choice Simple, internal, robust for polling tables and health checks Set timezone explicitly; do not rely on the self-hosted default
Webhook Yes, for instant downstream tasks Useful if the app or inference service should notify n8n immediately Good for report generation requests or post-inference fan-out
MQTT Trigger Yes, but only for low-frequency operational topics Works well for status or control-like events, not high-volume waveforms Do not subscribe n8n to raw pcg or ecg streams
Postgres Trigger Optional, later Can make database-driven workflows feel instant Heavier operational setup and trigger privileges need care with Supabase

Recommended implementation choice: start with Schedule Trigger plus Supabase reads/writes. Only introduce Webhook or Postgres Trigger once the first workflows are stable.

Suggested trigger mapping

Start with schedule-based polling

  • llm_reports every 1 minute
  • device health every 2 to 5 minutes
  • digest once daily
  • ops monitoring every 1 minute

Add webhooks later for speed

  • Queue a report in app -> call n8n webhook immediately
  • Inference completes -> call n8n webhook with session metadata
  • Alert opened -> trigger escalation chain immediately

7. Docker and Environment Plan

The current Compose file already runs Mosquitto, inference, frontend, and NGINX on a shared asculticor-network. n8n should join that network as another peer service with a persistent volume and explicit credentials for Supabase and any providers it needs to call. 

n8n:
  image: docker.n8n.io/n8nio/n8n:stable
  container_name: asculticor-n8n
  ports:
    - "127.0.0.1:5678:5678"
  environment:
    - N8N_BASIC_AUTH_ACTIVE=true
    - N8N_BASIC_AUTH_USER=${N8N_USER}
    - N8N_BASIC_AUTH_PASSWORD=${N8N_PASSWORD}
    - GENERIC_TIMEZONE=Africa/Cairo
    - ASCULTICOR_APP_URL=http://frontend:3000
    - ASCULTICOR_INFERENCE_URL=http://inference:8000
    - ASCULTICOR_INTERNAL_API_TOKEN=${INTERNAL_API_TOKEN}
    - ASCULTICOR_INFERENCE_TOKEN=${INFERENCE_INTERNAL_TOKEN}
    - SUPABASE_URL=${SUPABASE_URL}
    - SUPABASE_SERVICE_ROLE_KEY=${SUPABASE_SERVICE_ROLE_KEY}
    - MQTT_HOST=mosquitto
    - MQTT_PORT=1883
    - MQTT_USERNAME=${MQTT_USERNAME}
    - MQTT_PASSWORD=${MQTT_PASSWORD}
  volumes:
    - n8n_data:/home/node/.n8n
  networks:
    - asculticor-network
  restart: unless-stopped

Environment variables to add

  • N8N_USER
  • N8N_PASSWORD
  • GENERIC_TIMEZONE
  • Provider secrets such as OPENAI_API_KEY

Secrets to reuse safely

  • SUPABASE_URL
  • SUPABASE_SERVICE_ROLE_KEY
  • INTERNAL_API_TOKEN
  • INFERENCE_INTERNAL_TOKEN
  • MQTT_USERNAME and MQTT_PASSWORD for low-frequency topics only

If you later expose n8n webhooks publicly through NGINX, treat those routes as production API endpoints: protect them with header auth or another auth layer, and only expose the minimum set.

8. Recommended Rollout Roadmap

Phase 1: Infrastructure

  1. Add the n8n container to Docker Compose.
  2. Set persistent storage, basic auth, timezone, and provider credentials.
  3. Verify network connectivity from n8n to Supabase, frontend, inference, and Mosquitto.

Phase 2: Replace the LLM queue processor

  1. Keep the current route that inserts pending llm_reports rows.
  2. Build an n8n workflow that processes pending rows.
  3. Update status, retry, and completion inside the workflow.
  4. Once validated, retire the GitHub Actions cron and the report-generation logic in the route.

Phase 3: Add device-health alerting

  1. Poll devices and device_telemetry.
  2. Create device_alerts rows for offline and unhealthy states.
  3. Send Slack or email notifications.
  4. Add resolution and deduplication rules.

Phase 4: Reporting and escalation

  1. Build daily digest workflow.
  2. Add escalation logic for unresolved critical alerts.
  3. Optionally enrich historical summaries.

This phased order gives you one immediate business feature, one operational safety feature, and then a clean path toward richer automation.

9. Security and Operational Notes

Credentials and access

  • n8n will be powerful because it may hold the Supabase service-role secret.
  • Keep the n8n UI behind basic auth and preferably internal-only at first.
  • Use only the credentials each workflow truly needs.
  • Do not subscribe n8n broadly to raw medical waveform topics.

Workflow hygiene

  • Use idempotency keys or dedupe checks before creating alerts.
  • Use retry plus wait steps for transient provider failures.
  • Send structured fields back to Supabase, not only plain text.
  • Separate clinical-facing outputs from operational alert channels.

Medical product caution

Report and alert workflows must preserve the current disclaimer posture. n8n should not turn educational model outputs into deterministic diagnoses. It should automate handling, delivery, routing, and summarization of already-computed outputs.

Timezone note

Self-hosted n8n uses the instance timezone for schedules if a workflow timezone is not set. Set the instance timezone explicitly so daily digests and reminder workflows run at the expected local time.

10. What n8n Should Not Replace

Component Reason to keep it where it is
ESP32 firmware Hardware capture, sensor timing, and MQTT publishing stay on-device.
Mosquitto broker Handles device connectivity and message routing; n8n is not a broker replacement.
FastAPI MQTT handler Already built for binary chunk ingestion, buffering, live metrics, and session lifecycle.
ML preprocessing and inference These are compute-bound and tightly coupled to current model code.
Live waveform endpoints and frontend visualization Current implementation is tuned for polling plus SSE behavior tied to session views.
Device bootstrap and provisioning logic That is part of the current application trust boundary and device lifecycle.

n8n is not the right place for raw ECG or PCG chunks. It is a great place for "a row appeared", "a status changed", "a report is pending", or "an alert has not been resolved after 20 minutes".

Final Recommendation

Add n8n to AscultiCor, but add it in the narrowest, highest-leverage way first. Treat it as the async automation layer that lives after inference and around operations. Do not move ingestion, waveform handling, or model execution into it.

The strongest first implementation is:

  1. Deploy the n8n container.
  2. Build the real LLM report workflow.
  3. Replace the GitHub Actions scheduler.
  4. Add device-health alerts.
  5. Add digests and escalations once the first two workflows are stable.

This path gives you immediate user-visible improvement, reduces fragile custom async logic, and unlocks operational automation without disturbing the medical telemetry pipeline.
